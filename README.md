# Лабораторная работа по предмету "Бизнес логика программных систем"

## Задание

Вариант №2812: Деливери — быстрая доставка еды и продуктов — https://market-delivery.yandex.ru. Бизнес-процесс: формирование заказа и оформление доставки.

Описать бизнес-процесс в соответствии с нотацией BPMN 2.0, после чего реализовать его в виде приложения на базе Spring Boot.

Порядок выполнения работы:

Специфицировать модель реализуемого бизнес-процесса в соответствии с требованиями BPMN 2.0.

Разработать набор curl-скриптов, либо набор запросов для REST клиента Insomnia для тестирования публичных интерфейсов разработанного программного модуля. Запросы Insomnia оформить в виде файла экспорта.

| Параметр | Тип     | Обязательный | Описание             |

| username | string  | да           | Логин пользователя   |

```
POST /api/auth/login
Content-Type: application/json

{
    "username": "Harwyy",
```

### 21. Создать нового пользователя

Создаёт учётную запись с указанной ролью и (опционально) referenceId.

`POST /api/admin/users`

**Параметры запроса**

| Параметр     | Тип     | Обязательный | Описание                                                       |
|--------------|---------|--------------|----------------------------------------------------------------|
| username     | string  | да           | Логин                                                          |
| password     | string  | да           | Пароль                                                         |
| role         | string  | да           | Роль: ROLE_CUSTOMER, ROLE_RESTAURANT, ROLE_COURIER, ROLE_ADMIN |
| referenceId  | string  | нет          | ID связанной бизнес-сущности                                   |


**Пример запроса**

```
POST /api/admin/users
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{
    "username": "COURIER",
    "password": "COURIER",
    "role": "ROLE_COURIER",
    "referenceId": 1
}
```

**Ответ (200 OK)**

```json
{
  "username": "COURIER",
  "role": "ROLE_COURIER",
  "referenceId": 1
}
```  

###  Прецедент «Назначение курьера»

#### Название
Назначить курьера на готовый заказ

#### Актёр
Ресторан (роль ROLE_RESTAURANT или ROLE_ADMIN)

#### Предусловия

- Заказ находится в статусе READY
- В городе доставки заказа существует хотя бы один курьер со статусом AVAILABLE и количеством активных заказов < 2
- 
#### Основной поток

| Шаг | Действие                                                                                             |
|-----|------------------------------------------------------------------------------------------------------|
| 1   | Система получает orderId заказа в статусе READY                                                      |
| 2   | Загружает заказ из БД                                                                                |
| 3   | Определяет город доставки заказа                                                                     |
| 4   | Находит всех курьеров в этом городе со статусом AVAILABLE, у которых количество активных заказов < 2 |
| 5   | Для каждого подходящего курьера вычисляет рейтинг                                                    |
| 6   | Выбирает курьера с наивысшим рейтингом                                                               |
| 7   | Изменяет статус выбранного курьера на BUSY и увеличивает счётчик активных заказов на 1               |
| 8   | Привязывает курьера к заказу                                                                         |
| 9   | Меняет статус заказа на ASSIGNED и фиксирует время назначения                                        |
| 10  | Сохраняет изменения в БД                                                                             |
| 11  | Фиксирует транзакцию                                                                                 |

#### Альтернативные потоки

- Нет доступных курьеров (в городе нет курьеров с AVAILABLE и загрузкой < 2) - транзакция откатывается
-Заказ не в статусе READY - транзакция откатывается

- Каждый заказ в списке принадлежит данному ресторану
- Все заказы находятся в статусе READY
- Для каждого заказа есть доступные курьеры в городе доставки

#### Основной поток

| Шаг | Действие                                                                                                                  |

| 1   | Ресторан отправляет запрос POST /api/restaurants/{restaurantId}/batch/assign-couriers с телом, содержащим массив orderIds |
| 2   | Система начинает транзакцию                                                                                               |
| 3   | Для каждого orderId:                                                                                                      |
| 3.2 | Загружает заказ и проверяет, что он принадлежит ресторану с restaurantId                                                  |
| 3.3 | Проверяет, что статус заказа равен READY. Если нет – выбрасывает BusinessException                                        |
| 3.4 | Привязывает курьера                                                                                                       |
| 3.5 | Добавляет в список ответов запись с orderId, новым статусом ASSIGNED и сообщением "Курьер назначен"                       |

###  Прецедент «Создание заказа»

#### Название
Создание заказа с проверкой доступности, оплатой и расчётом времени доставки

#### Актёр
Клиент (роль ROLE_CUSTOMER или ROLE_ADMIN)

#### Предусловия

- Ресторан с указанным restaurantId существует и активен
- Все переданные продукты существуют, доступны (available = true) и принадлежат данному ресторану
- У клиента либо есть сохранённый адрес, либо в запросе передан адрес доставки
- Клиент аутентифицирован

#### Основной поток

| Шаг | Действие                                                                                                                                                                                        |
|-----|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1   | Система получает запрос POST /api/orders                                                                                                                                                        |
| 2   | Проверяет, что список позиций (items) не пуст. Если пуст – выбрасывает BusinessException (откат транзакции)                                                                                     |
| 3   | Загружает пользователя по userId                                                                                                                                                                |
| 4   | Загружает ресторан по restaurantId                                                                                                                                                              |
| 5   | Определяет адрес доставки: если в запросе передан адрес с заполненным городом – сохраняет его; иначе берёт адрес из профиля пользователя. Если нигде нет адреса – выбрасывает BusinessException |
| 6   | Создаёт объект Order со статусом CREATED, привязывает пользователя, ресторан, адрес доставки, комментарии                                                                                       |
| 7   | Для каждой позиции из запроса                                                                                                                                                                   |
| 7.1 | Загружает продукт по productId                                                                                                                                                                  |
| 7.2 | Проверяет, что продукт доступен (available = true). Иначе BusinessException                                                                                                                     |
| 7.3 | Проверяет, что продукт принадлежит указанному ресторану                                                                                                                                         |
| 7.4 | Создаёт OrderItem с количеством и ценой продукта                                                                                                                                                |
| 7.5 | Накапливает общую стоимость заказа                                                                                                                                                              |
| 8   | Назначает заказу итоговую сумму и список позиций                                                                                                                                                |
| 9   | Сохраняет заказ в БД                                                                                                                                                                            |
| 10  | Проверяет, входит ли адрес доставки в зону доставки ресторана                                                                                                                                   |
| 11  | Если адрес вне зоны доставки: переводит заказ в статус CANCELLED, сохраняет, и возвращает ответ (транзакция фиксируется)                                                                        |
| 12  | Если адрес в зоне доставки: меняет статус на WAITING_PAYMENT                                                                                                                                    |
| 13  | Вызывает платёжную систему                                                                                                                                                                      |
| 14  | Вычисляет ориентировочное время доставки и сохраняет его в заказе                                                                                                                               |
| 15  | Транзакция фиксируется. Возвращает OrderResponse с итоговым состоянием заказа                                                                                                                   |

#### Альтернативные потоки

- Пустой список позиций - исключение BusinessException, откат транзакции
- Продукт недоступен или не принадлежит ресторану - исключение BusinessException, откат транзакции
- Пользователь или ресторан не найдены → исключение ResourceNotFoundException, откат транзакции
- Отсутствует адрес доставки → исключение BusinessException, откат транзакции

###  Прецедент «Отмена оплаченного заказа с частичным возвратом средств»

#### Название
Отмена оплаченного заказа с частичным возвратом средств

#### Актёр
Клиент

#### Предусловия

- Клиент аутентифицирован и имеет роль CUSTOMER.
- Заказ с указанным orderId существует в системе.
- Заказ принадлежит данному клиенту

#### Основной поток

| Шаг | Действие                                                                                                   |
|-----|------------------------------------------------------------------------------------------------------------|
| 1   | Клиент отправляет запрос POST /api/orders/{orderId}/cancel с JWT-токеном в заголовке                       |
| 2   | Система проверяет JWT и извлекает customerId                                                               |
| 3   | Система находит заказ по orderId. Если заказ не найден → исключение ResourceNotFoundException              |
| 4   | Система проверяет, что заказ принадлежит клиенту                                                           |
| 5   | Система определяет текущий статус заказа и вычисляет процент возврата                                      |
| 6   | Если процент возврата равен 0, система выбрасывает BusinessException с сообщением о невозможности возврата |
| 7   | Система вызывает сервис возврата средств                                                                   |
| 8   | В случае успеха система переводит заказ в статус CANCELLED и сохраняет изменения                           |
| 9   | Система возвращает клиенту обновлённый OrderResponse с новым статусом                                      |

#### Альтернативные потоки

- Недостаточно прав (роль не CUSTOMER) - 403 Forbidden
- Заказ не принадлежит клиенту - BusinessException
- Возврат невозможен по статусу - BusinessException
- Ошибка платёжной системы - транзакция откатывается, статус заказа не меняется

## Роли и привилегии

| Роль            | Описание                                                                                         |
|-----------------|--------------------------------------------------------------------------------------------------|
| ROLE_CUSTOMER   | Клиент сервиса. Просматривает рестораны, меню, создаёт заказы.                                   |
| ROLE_RESTAURANT | Владелец/менеджер ресторана. Управляет заказами своего ресторана.                                |
| ROLE_COURIER    | Курьер. Работает с заказами, назначенными ему.                                                   |
| ROLE_ADMIN      | Администратор системы. Имеет полный доступ ко всем операциям без ограничений по принадлежности.  |
