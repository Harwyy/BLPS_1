# Лабораторная работа по предмету "Бизнес логика программных систем"

## Задание

Вариант №2812: Деливери — быстрая доставка еды и продуктов — https://market-delivery.yandex.ru. Бизнес-процесс: формирование заказа и оформление доставки.

Описать бизнес-процесс в соответствии с нотацией BPMN 2.0, после чего реализовать его в виде приложения на базе Spring Boot.

Порядок выполнения работы:

Специфицировать модель реализуемого бизнес-процесса в соответствии с требованиями BPMN 2.0.

Разработать набор curl-скриптов, либо набор запросов для REST клиента Insomnia для тестирования публичных интерфейсов разработанного программного модуля. Запросы Insomnia оформить в виде файла экспорта.

| Параметр | Тип     | Обязательный | Описание             |

| username | string  | да           | Логин пользователя   |

```
POST /api/auth/login
Content-Type: application/json

{
    "username": "Harwyy",
```

### 21. Создать нового пользователя

Создаёт учётную запись с указанной ролью и (опционально) referenceId.

`POST /api/admin/users`

**Параметры запроса**

| Параметр     | Тип     | Обязательный | Описание                                                       |
|--------------|---------|--------------|----------------------------------------------------------------|
| username     | string  | да           | Логин                                                          |
| password     | string  | да           | Пароль                                                         |
| role         | string  | да           | Роль: ROLE_CUSTOMER, ROLE_RESTAURANT, ROLE_COURIER, ROLE_ADMIN |
| referenceId  | string  | нет          | ID связанной бизнес-сущности                                   |


**Пример запроса**

```
POST /api/admin/users
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{
    "username": "COURIER",
    "password": "COURIER",
    "role": "ROLE_COURIER",
    "referenceId": 1
}
```

**Ответ (200 OK)**

```json
{
  "username": "COURIER",
  "role": "ROLE_COURIER",
  "referenceId": 1
}
```  

###  Прецедент «Назначение курьера»

#### Название
Назначить курьера на готовый заказ

#### Актёр
Система (автоматический процесс после отметки «заказ готов»)

#### Предусловия

Заказ находится в статусе READY (приготовлен рестораном). Существует хотя бы один курьер в том же городе со статусом AVAILABLE и количеством активных заказов < 2.

#### Основной поток

| Шаг | Действие                                                                        |
|-----|---------------------------------------------------------------------------------|
| 1   | Система получает id заказа в статусе READY                                      |
| 2   | Загружает данные заказа                                                         |
| 3   | Находит всех курьеров в том же городе со статусом AVAILABLE и загрузкой < 2     |
| 4   | Для каждого курьера вычисляет рейтинг                                           |
| 5   | Выбирает курьера с наивысшим рейтингом                                          |
| 6   | Меняет статус выбранного курьера на BUSY и увеличивает счётчик активных заказов |
| 7   | Привязывает курьера к заказу                                                    |
| 8   | Меняет статус заказа на ASSIGNED и фиксирует время назначения                   |
| 9   | Сохраняет все изменения в БД                                                    |
| 10  | Фиксирует транзакцию                                                            |

#### Альтернативные потоки

- Курьер не найден – транзакция откатывается, заказ остаётся в статусе READY, система выбрасывает исключение «Нет доступных курьеров».
- Заказ не в статусе READY – транзакция откатывается, исключение «Заказ можно назначить только в статусе READY».
- Ошибка при сохранении (сбой БД) – транзакция откатывается, заказ и курьер не изменяются.

###  Прецедент «Массовый перевод заказов в статус «Готов»»

#### Название
Пакетная отметка готовности заказов

#### Актёр
Ресторан

#### Предусловия

- Передан список идентификаторов заказов, принадлежащих этому ресторану.
- Каждый заказ в списке находится в статусе PREPARING (готовится).

#### Основной поток

| Шаг | Действие                                                             |
|-----|----------------------------------------------------------------------|
| 1   | Система получает список идентификаторов заказов и restaurantId       |
| 2   | Для каждого orderId в списке:                                        |
| 2.1 | Загружает заказ и проверяет, что он принадлежит указанному ресторану |
| 2.2 | Проверяет, что статус заказа = PREPARING                             |
| 2.3 | Выбирает курьера с наивысшим рейтингом                               |
| 2.4 | Меняет статус заказа на READY и сохраняет                            |
| 3   | Если все заказы обработаны успешно – фиксирует транзакцию            |
| 4   | Возвращает список ответов с новым статусом каждого заказа            |

#### Альтернативные потоки

- Если хотя бы один заказ не найден, не принадлежит ресторану или имеет статус не PREPARING - транзакция полностью откатывается, ни один заказ не меняет статус

- Для каждого заказа существует хотя бы один доступный курьер (город совпадает, статус AVAILABLE, активных заказов < 2).

#### Основной поток

| Шаг  | Действие                                                                    |
|------|-----------------------------------------------------------------------------|
| 1    | Система получает список идентификаторов заказов и restaurantId              |
| 2    | Для каждого orderId в списке:                                               |
| 2.1  | Загружает заказ и проверяет, что он принадлежит указанному ресторану        |
| 2.2  | Проверяет, что статус заказа = READY                                        |
| 2.3  | Находит всех курьеров в том же городе со статусом AVAILABLE и загрузкой < 2 |
| 2.4  | Для каждого курьера вычисляет рейтинг                                       |
| 2.5  | Выбирает курьера с наивысшим рейтингом                                      |
| 2.6  | Меняет статус курьера на BUSY, увеличивает счётчик активных заказов         |

| 2.9  | Сохраняет изменения                                                         |

###  Прецедент «Создание заказа»

#### Название
Создание заказа с оплатой и проверкой доступности

#### Актёр
Клиент

#### Предусловия

- Ресторан существует и активен.
- Продукты, указанные в заказе, существуют, доступны и принадлежат выбранному ресторану.
- Адрес доставки указан (либо у пользователя есть адрес, либо передан в запросе)

#### Основной поток

| Шаг | Действие                                                                                                |
|-----|---------------------------------------------------------------------------------------------------------|
| 1   | Система получает запрос на создание заказа (пользователь, ресторан, список позиций, адрес, комментарии) |
| 2   | Проверяет, что список позиций не пуст                                                                   |
| 3   | Загружает пользователя и ресторан                                                                       |
| 4   | Определяет адрес доставки (из запроса или из профиля пользователя)                                      |
| 5   | Создаёт новый заказ со статусом CREATED                                                                 |
| 6   | Для каждой позиции:                                                                                     |
| 6.1 | Проверяет доступность продукта (available = true)                                                       |
| 6.2 | Проверяет, что продукт принадлежит указанному ресторану                                                 |
| 6.3 | Создаёт OrderItem с количеством и ценой                                                                 |
| 6.4 | Накапливает общую сумму заказа                                                                          |
| 7   | Устанавливает общую сумму и список позиций в заказе                                                     |
| 8   | Сохраняет заказ                                                                                         |
| 9   | Проверяет, входит ли адрес доставки в зону доставки ресторана                                           |
| 10  | Если расстояние превышает допустимое – устанавливает статус CANCELLED и завершает транзакцию            |
| 11  | Если расстояние в пределах зоны – переводит заказ в статус WAITING_PAYMENT                              |
| 12  | Вызывает внешнюю платёжную систему                                                                      |
| 13  | Если оплата не прошла – переводит заказ в статус CANCELLED, выбрасывает исключение (откат транзакции)   |
| 14  | Если оплата успешна – переводит заказ в статус PAID                                                     |
| 15  | Рассчитывает оценочное время доставки, сохраняет в заказе                                               |
| 16  | Фиксирует транзакцию                                                                                    |
| 16  | Возвращает DTO заказа клиенту                                                                           | 

#### Альтернативные потоки

- Пустой список позиций → исключение, откат
- Продукт недоступен или не принадлежит ресторану → исключение, откат
- Нет адреса доставки → исключение, откат
- Адрес вне зоны доставки → заказ отменяется (статус CANCELLED), но транзакция фиксируется. В этом случае оплата не вызывается

## Роли и привилегии

| Роль            | Описание                                                                                         |
|-----------------|--------------------------------------------------------------------------------------------------|
| ROLE_CUSTOMER   | Клиент сервиса. Просматривает рестораны, меню, создаёт заказы.                                   |
